# Consistent Hashing

## 문제의 시작: 서버가 여러 대일 때 데이터를 어디에 저장할까?

캐시 서버가 4대 있다고 해보자. 사용자의 요청을 어떤 서버로 보내야 할까?

가장 단순한 방법은 이거다:

```
서버 번호 = hash(key) % 서버 수
```

예를 들어 서버가 4대면:

```
hash("user:123") % 4 = 2  → 서버 2에 저장
hash("user:456") % 4 = 0  → 서버 0에 저장
hash("user:789") % 4 = 3  → 서버 3에 저장
```

간단하고 잘 동작한다. **서버 수가 변하지 않는다면.**

---

## 단순 해시의 치명적인 문제

### 서버가 1대 추가되면?

서버가 4대에서 5대로 늘어났다고 하자. `% 4`가 `% 5`로 바뀐다.

```
변경 전 (서버 4대):                변경 후 (서버 5대):
hash("user:123") % 4 = 2          hash("user:123") % 5 = 3  ← 변경!
hash("user:456") % 4 = 0          hash("user:456") % 5 = 1  ← 변경!
hash("user:789") % 4 = 3          hash("user:789") % 5 = 4  ← 변경!
```

거의 모든 키의 서버 할당이 바뀐다. 캐시 서버라면? **기존 캐시가 전부 무효화**된다. 서버 1대 추가했을 뿐인데, 모든 요청이 cache miss를 일으키면서 원본 DB로 몰린다. 이걸 **cache stampede**라고 한다. 서비스가 다운될 수 있다.

수치로 보면:

| 서버 수 변화 | 재배치되는 키 비율 |
|-------------|-------------------|
| 4 → 5 | **약 80%** |
| 10 → 11 | **약 90%** |
| 100 → 101 | **약 99%** |

서버가 많을수록 상황이 더 나빠진다. 서버 수가 바뀔 때마다 거의 전체 데이터가 이사를 가야 한다.

### 서버가 1대 죽으면?

더 심각한 문제다. 서버가 죽는 건 계획 없이 갑자기 일어난다. 서버 4대 중 1대가 죽으면 `% 4`가 `% 3`이 되면서 역시 대부분의 키가 재배치된다. 장애 상황에서 cache stampede까지 겹치면 완전히 끝난다.

**핵심 문제: 서버 수가 변하면, 변한 서버와 무관한 키까지 전부 영향을 받는다.**

## Consistent Hashing의 핵심 아이디어

1997년 MIT의 David Karger가 제안한 알고리즘이다. 핵심 아이디어는 놀라울 정도로 단순하다.

### 해시 공간을 원형 링으로 만든다

보통 해시 함수의 출력은 직선적인 숫자 범위다 (예: 0 ~ 2^32). 이 직선의 양 끝을 이어서 **원(ring)**으로 만든다고 상상하자.

```
        0 (= 2^32)
        │
   ─────┼─────
  │           │
  │   Hash    │
  │   Ring    │
  │           │
   ─────┼─────
        │
      2^31
```

### 서버를 링 위에 배치한다

각 서버의 이름(또는 IP)을 해시해서 링 위의 한 지점에 놓는다.

```
          0
          │
     S3 ──┼── S1
    ╱           ╲
   │             │
    ╲           ╱
     S2 ──┼──
          │
```

### 키도 링 위에 배치하고, 시계 방향으로 가장 가까운 서버에 저장한다

데이터 키를 해시해서 링 위에 놓은 뒤, **시계 방향으로 이동하다 처음 만나는 서버**에 저장한다.

```
          0
          │
     S3 ──┼── S1
    ╱  k3↗  k1↗  ╲
   │               │
    ╲   k2↗      ╱
     S2 ──┼──
          │

k1 → 시계 방향 → S1에 저장
k2 → 시계 방향 → S2에 저장  (S2를 지나서)
k3 → 시계 방향 → S3에 저장
```

### 서버가 추가/제거되면?

**서버 추가 (S4가 S2와 S3 사이에 들어옴):**

```
          0
          │
     S3 ──┼── S1
    ╱           ╲
   S4             │
    ╲           ╱
     S2 ──┼──
          │
```

S4가 들어오면, **S4와 반시계 방향의 이전 서버(S2) 사이에 있던 키들만** S4로 이동한다. 나머지 키들은 그대로다.

**서버 제거 (S2가 죽음):**

S2에 있던 키들만 시계 방향의 다음 서버(S3)로 이동한다. 나머지는 그대로다.

이게 Consistent Hashing의 핵심이다. **서버가 변해도, 영향 받는 건 인접한 구간의 키뿐이다.**

수치로 비교하면:

| | 단순 해시 (% N) | Consistent Hashing |
|---|---|---|
| 서버 4→5 추가 | **~80% 재배치** | **~20% 재배치** (1/N) |
| 서버 10→11 추가 | **~90% 재배치** | **~9% 재배치** |
| 서버 100→101 추가 | **~99% 재배치** | **~1% 재배치** |

서버가 많을수록 Consistent Hashing의 이점이 극적으로 커진다.

## 원형 링은 실제로 어떻게 구현하나?

실제로 원형 자료구조를 만드는 게 아니다. 정렬된 맵(TreeMap)에 **"못 찾으면 처음으로 돌아간다"는 규칙 하나**만 추가하면 논리적으로 원이 된다.

시계를 생각하면 된다. 12시 다음은 1시지, 13시가 아니다. 해시 링도 마찬가지로 2^32 다음은 다시 0이다.

```kotlin
fun getNode(key: String): String? {
    val hash = hash(key)
    
    // 시계 방향으로 가장 가까운 서버를 찾는다
    val entry = ring.ceilingEntry(hash)
    
    // 못 찾으면? → 링의 끝을 지난 것 → 처음으로 돌아간다
    return entry?.value ?: ring.firstEntry().value
    //                      ↑ 이 한 줄이 "원형"을 만든다
}
```

구체적인 예시:

```
링 위의 서버 위치 (해시값):
  S1 = 1000, S2 = 5000, S3 = 9000
  해시 범위: 0 ~ 10000

key의 해시가 7000이면?
→ ceilingEntry(7000) = S3(9000) ✓

key의 해시가 9500이면?
→ ceilingEntry(9500) = null (9000 뒤에 서버 없음)
→ firstEntry() = S1(1000)  ← 원형이니까 처음으로!
```

## 트래픽은 어떻게 분배되나?

로드밸런서(또는 클라이언트 라이브러리)가 매 요청마다 해시 링을 탐색해서 담당 서버를 결정한다.

```
1. "user:123" 데이터 요청이 온다
2. hash("user:123") = 7200
3. 링에서 7200 이상인 첫 번째 서버 = S3(9000)
4. 요청을 S3으로 보낸다
```

**같은 키는 항상 같은 서버로 간다.** `hash("user:123")`은 항상 7200이고, 시계 방향 다음은 항상 S3이니까. 캐시에서 이게 중요한 이유는, 같은 데이터가 항상 같은 캐시 서버에 있어야 cache hit이 나기 때문이다.

### 라운드 로빈과의 차이

```
라운드 로빈:
요청 1 → S1, 요청 2 → S2, 요청 3 → S3 (순서대로 돌림)
→ 같은 사용자가 매번 다른 서버로 감 → 캐시 의미 없음

Consistent Hashing:
hash("user:123") → 항상 S3
hash("user:456") → 항상 S1
→ 같은 키는 항상 같은 서버 → 캐시 히트율 극대화
```

### 누가 이 결정을 하나?

```
방식 1 — 로드밸런서가 결정:
Client → [Nginx: hash(key) → 서버 결정] → Server A/B/C

방식 2 — 클라이언트 라이브러리가 직접 결정:
Client (Redis SDK가 hash(key) → 노드 결정) → Redis Node A/B/C
```

Redis Cluster가 방식 2다. 클라이언트 라이브러리가 키를 해싱해서 어떤 노드에 직접 연결할지 스스로 결정한다.

## 서버 추가 시 실제로 무슨 일이 일어나나?

S11이 S3과 S4 사이에 등록되면, **기존 S4가 담당하던 키 중 일부가 자동으로 S11으로 간다.**

```
변경 전:
S3(3000) ──── key:A(5000), key:B(6000), key:C(7500) ──── S4(8000)
              이 키들 전부 S4가 담당

S11 등록: hash("S11") = 6500

변경 후:
S3(3000) ── key:A(5000), key:B(6000) ── S11(6500) ── key:C(7500) ── S4(8000)
            이제 S11 담당                              여전히 S4 담당
```

별도로 "이제부터 너 담당이야"라고 알려주는 게 아니다. **링에 S11을 등록하는 순간, 로드밸런서의 해시 탐색 결과가 자동으로 바뀐다.** 기존 서버(S4)는 아무것도 안 한다.

### 영향 범위는?

```
링 정보 업데이트: 모든 로드밸런서/클라이언트가 받긴 함
실제 트래픽 변화: S3~S4 구간 → S11로 넘어가는 부분만
나머지 S1~S3, S5~S10: 요청 흐름 변화 없음
```

### Cache Miss는 어떻게 되나?

S11은 빈 서버로 시작하니까 처음에는 전부 cache miss다. S4에 캐시가 있는데 이제 S11한테 물어보니까 miss가 나는 것이다.

```
변경 전: "key:A 줘" → S4 → "있다!" (hit)
변경 후: "key:A 줘" → S11 → "나 방금 왔는데 없어" (miss) → DB에서 가져옴
         한편 S4에는 key:A가 아직 남아있음 (아무도 안 찾아감, TTL 후 삭제)
```

Consistent Hashing이 cache miss를 **0으로 만드는 건 불가능**하다. **1/N으로 최소화**하는 것이다. 대부분의 경우 이 정도의 miss는 DB가 충분히 감당할 수 있다.

### Cold Start 대응 전략

| 방식 | 설명 | 적합한 경우 |
|------|------|-------------|
| **그냥 둔다** | miss 나면 DB에서 가져오면서 자연스럽게 캐시 채움 | 대부분의 서비스 (가장 일반적) |
| **Warm-up** | 투입 전 기존 서버에서 데이터를 미리 복사 | 트래픽이 매우 큰 서비스 |
| **점진적 전환** | 가상 노드를 조금씩 추가해서 트래픽을 서서히 유입 | Netflix, Discord 급 |

## 문제점: 불균등 분배

여기까지만 보면 완벽해 보이지만, 실제로는 심각한 문제가 있다.

서버가 3대인데 해시 값이 이렇게 나왔다고 하자:

```
          0
          │
          ┼── S1 (위치: 10)
          │
          │
          │
          │
          │
          │
     S3 ──┼── S2 (위치: 190)
   (위치: 200)
```

S1은 링의 아주 작은 구간만 담당하고, S2와 S3는 넓은 구간을 담당한다. 트래픽이 균등하게 분배되지 않는다. 서버가 적을수록 이 문제가 심해진다.

운이 나쁘면 한 서버에 트래픽의 70~80%가 몰릴 수 있다. 이러면 그 서버만 과부하에 걸린다.

---

## 해결: Virtual Node (가상 노드)

이 문제의 해결책이 **Virtual Node**다. 물리 서버 하나를 링 위에 **여러 개의 가상 노드**로 복제하는 것이다.

서버 3대에 각각 가상 노드 4개씩이면, 링 위에 총 12개 지점이 생긴다:

```
S1 → S1-0, S1-1, S1-2, S1-3  (링 위에 4개 지점)
S2 → S2-0, S2-1, S2-2, S2-3  (링 위에 4개 지점)
S3 → S3-0, S3-1, S3-2, S3-3  (링 위에 4개 지점)
```

```
           0
     S2-1 ─┼─ S1-0
    ╱    S3-2    ╲
  S1-2          S3-0
  S3-1          S2-2
    ╲    S1-3    ╱
     S2-0 ─┼─ S1-1
```

각 서버의 가상 노드가 링 전체에 고르게 퍼지면서, 담당 구간이 자연스럽게 균등해진다.

**가상 노드가 많을수록 분배가 균등해진다:**

| 가상 노드 수 | 부하 표준편차 |
|-------------|-------------|
| 1 (가상 노드 없음) | ~60% |
| 100 | ~10% |
| 200 | ~5% |
| 500+ | ~2% 이하 |

실무에서는 보통 **100~200개**의 가상 노드를 쓴다.

**가상 노드의 해시는 어떻게 만드는가?**

```
hash("S1-0"), hash("S1-1"), hash("S1-2"), ...
```

서버 이름에 인덱스를 붙여서 해시한다. 이렇게 하면 같은 서버의 가상 노드들이 링 위에 랜덤하게 분산된다.

**Trade-off:** 가상 노드가 많으면 분배는 균등해지지만, 키를 찾을 때 탐색해야 할 노드 수가 늘어난다. 하지만 정렬된 배열에서 이진 탐색(O(log N))을 쓰면, 가상 노드 수천 개도 문제없다.

## 서버 간 데이터 복제는?

실무에서 Consistent Hashing을 쓸 때는 보통 **데이터를 여러 서버에 복제**한다. 키가 할당된 서버(primary) 외에, 시계 방향으로 다음 N-1개 서버에도 복사본을 둔다.

```
복제 계수 N = 3인 경우:

k1 → S1 (primary)
   → S2 (replica 1, 시계 방향 다음)
   → S3 (replica 2, 그 다음)
```

S1이 죽어도 S2, S3에 데이터가 있으니 서비스가 계속된다. Amazon DynamoDB가 정확히 이 방식을 쓴다.

주의할 점: 가상 노드를 쓸 때, 시계 방향 다음 노드가 **같은 물리 서버의 가상 노드**일 수 있다. 이 경우 건너뛰어야 진짜 다른 서버에 복제된다.

## 실제 어디서 쓰이나?

| 시스템 | 용도 |
|--------|------|
| **Amazon DynamoDB** | 파티션 키로 데이터 분산 + 복제 |
| **Apache Cassandra** | 링 기반 데이터 파티셔닝 |
| **Redis Cluster** | 16384개 해시 슬롯을 노드에 분배 (변형) |
| **Akamai CDN** | 콘텐츠를 엣지 서버에 분배 (최초 발명 동기) |
| **Discord** | 수억 사용자의 메시지를 서버에 분산 |
| **Nginx (upstream)** | `hash ... consistent` 로드밸런싱 |

Rate Limiter에서 Redis를 중앙 저장소로 쓴다고 했는데, Redis가 한 대로 부족해지면 Redis Cluster를 쓰게 된다. 이때 "이 키는 어떤 Redis 노드에 있는가"를 결정하는 게 바로 Consistent Hashing의 변형이다.

## Redis Cluster는 정확히 Consistent Hashing인가?

엄밀히 말하면 아니다. Redis Cluster는 **Hash Slot** 방식을 쓴다.

```
CRC16(key) % 16384 = slot 번호
```

16384개의 고정 슬롯을 만들어 놓고, 각 노드에 슬롯 범위를 할당한다:

```
노드 A: 슬롯 0 ~ 5460
노드 B: 슬롯 5461 ~ 10922
노드 C: 슬롯 10923 ~ 16383
```

노드를 추가하면? 기존 노드들에서 슬롯 일부를 떼어서 새 노드에 넘긴다. 키 자체의 해시는 바뀌지 않고, **슬롯의 소유권만 이전**된다.

Consistent Hashing과 같은 문제(최소한의 재배치)를 풀지만, 구현은 더 단순하고 예측 가능하다. 어떤 키가 어느 노드에 있는지 슬롯 테이블만 보면 바로 알 수 있다.

## 구간의 크기 = 트래픽 양

링 위에서 각 서버가 담당하는 구간이 넓으면, 그만큼 많은 키의 해시값이 그 안에 떨어진다. **구간이 넓은 서버 = 트래픽을 많이 받는 서버**다.

```
S1(1000) ─── S2(3000) ─────────────────── S3(9000) ─── S1로 돌아감
  │← 2000 →│←         6000           →│←  2000  →│

S2 담당 구간: 2000 (전체의 20%)
S3 담당 구간: 6000 (전체의 60%)  ← S3에 트래픽 60% 몰림!
S1 담당 구간: 2000 (전체의 20%)
```

이걸 균등하게 만드는 게 Virtual Node다. 큰 구간 하나 대신 **작은 구간 100개를 여기저기 담당**하면, 주사위를 300번 던지면 각 눈이 1/6에 수렴하는 것처럼, 전체적으로 균등해진다.

### 특정 서버에 트래픽이 몰리면?

| 상황 | 원인 | 해결 |
|------|------|------|
| 한 서버만 바쁨 | 구간 불균등 | Virtual Node 수 늘리기 |
| 한 서버만 바쁨 | Hot Key (특정 키에 요청 폭주) | 키 분산, 로컬 캐시, 복제 |
| 모든 서버가 바쁨 | 전체 트래픽 증가 | 서버 추가 |

**Hot Key 문제**는 Consistent Hashing으로 해결이 안 된다. 예를 들어 연예인 스캔들 뉴스 하나에 초당 100만 요청이 몰리면, 해시값은 안 바뀌니까 같은 서버로만 간다. 이때는:

```
방법 1: 로컬 캐시 — 각 서버가 핫 키를 자기 메모리에도 캐싱
방법 2: 키 분산 — "news:12345" → "news:12345_r{0~9}"로 10개 서버에 분산
방법 3: 읽기 복제 — 핫 키를 여러 서버에 복사하고 요청을 랜덤 분산
```

---

## 여담: 안드로이드 개발자가 왜 이걸 알아야 하나?

직접 Consistent Hashing을 구현할 일은 없다. 하지만 이 개념을 알면:

1. **캐시 전략 이해**: Glide/Coil 같은 이미지 라이브러리도 내부적으로 캐시 키를 해싱한다. 캐시 무효화가 왜 어려운지 체감할 수 있다.

2. **백엔드와 소통**: "서버 증설했더니 캐시 히트율이 떨어졌어요"라는 상황에서 왜 그런지 바로 이해할 수 있다. Consistent Hashing을 안 쓰고 있다면 제안할 수도 있다.

3. **면접**: 시스템 디자인 면접에서 "서버를 어떻게 스케일하겠느냐"는 질문의 기본 답변이다. Uber, Twitter, Instagram 등 거의 모든 대규모 서비스 설계에 등장한다.

4. **Rate Limiter 연결**: 01에서 공부한 분산 Rate Limiter가 Redis Cluster를 쓴다면, 그 Redis Cluster 안에서 Consistent Hashing이 동작하고 있는 것이다.
